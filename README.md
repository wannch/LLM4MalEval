# LLM4MalEval

This repo provides the code, datasets, evaluation scripts, and experimental results of the LLM4MalEval Benchmark.

## Code

- `main.py`: the main experiment runner. It loads benchmark datasets, queries the selected LLM, and writes raw model outputs under `results/`.
- `llms/`: LLMs access and prompt used, including Llama, CodeLlama, Qwen, and DeepSeek models.
- `inject/`: AST-based code injection utilities. It implements seven code-mixture strategies. See `inject/README.md` for details.
- `obfuscate/`: code obfuscation implem of four obfuscation levels. See `obfuscate/README.md` for details.
- `annotate_sample.py`: automatic annotation and manual verification for LLM outputs.
- `statistics_*.py`: scripts for aggregating annotated results into evaluation metrics and CSV summaries.

## Dataset

The `malicious_packages_benchmark/` directory contains the benchmark data used by the real-world, obfuscation, and code-mixture experiments.

Important files and folders include:

- `malicious_code_info_merge_with_typelabel.json`: metadata and code content for real-world malicious samples.
- `benign_pyfiles`: benign code files.
- `benign_code_info.json`: metadata for benign samples.
- `obfuscated_samples_info.json`: metadata for obfuscated benign and malicious samples.
- `inject_samples_info.json`: metadata for code-mixture samples.
- `malicious_code_384_by_category/`: categorized malicious code samples.
- `benign_65/`: selected benign Python files.
- `benign_65_obfuscated/`: obfuscated benign samples.
- `malicious_inject_2_benign/`: code-mixture samples generated by injection.

## Results

- `results/`: raw LLM outputs produced by `main.py`.
- `results_annotation/`: GPT-5.2 annotation results for the raw LLM outputs.
- `results_evaluate_metric/`: aggregated metrics and CSV files generated from annotated outputs.

The raw result directory structure follows the experiment setting and prompt mode, for example:

```text
results/real_world/malicious_samples/zero-shot/
results/real_world/benign_samples/zero-shot/
results/obfuscation/zero-shot/
results/code_mixture/zero-shot/
```

## Running Experiments

Run experiments with `main.py` from the repository root:

```bash
python main.py --RQ <scenario> --model <model_name_or_path> --recmode <True|False> --few-shot <True|False>
```

Arguments:

- `--RQ`: experiment scenario. Supported values are `realworld_malicious`, `realworld_benign`, `obfuscation`, and `codemixture`.
- `--model`: model alias, Hugging Face model name, or local model path supported by the wrappers in `llms/`.
- `--few-shot`: whether to use the in-context few-shot prompt. Use `True` or `False`.

Supported models:

```text
qwen2-7b-instruct
qwen2-72b-instruct
qwen3-8b
qwen3-14b
qwen3-32b
deepseek-v3
deepseek-r1
meta-llama/Llama-2-7b-chat-hf
meta-llama/Llama-2-13b-chat-hf
meta-llama/Llama-2-70b-chat-hf
meta-llama/Meta-Llama-3-8B-Instruct
meta-llama/Meta-Llama-3-70B-Instruct
codellama/CodeLlama-7b-Instruct-hf
codellama/CodeLlama-13b-Instruct-hf
codellama/CodeLlama-34b-Instruct-hf
meta-llama/CodeLlama-70b-Instruct-hf
```

The model wrappers load local Hugging Face models with `transformers`. Large models require enough GPU memory or an external deployment runtime. Llama-family models also require Hugging Face access permissions.

### Real-World Malicious Samples

This scenario evaluates whether an LLM detects real-world malicious package code.

```bash
python main.py \
  --RQ realworld_malicious \
  --model qwen3-8b \
  --few-shot False
```

Output:

```text
results/real_world/malicious_samples/zero-shot/qwen3-8b.json
```

Few-shot run:

```bash
python main.py \
  --RQ realworld_malicious \
  --model qwen3-8b \
  --few-shot True
```

## Running Automated Verification

`annotate_sample.py` verifies raw LLM outputs by asking GPT-5.2 to judge each response against five metric dimensions.

The five annotation dimensions are:

- `VALID_RESPONSE`: whether the LLM output is a valid response for malicious-code identification.
- `EXPLICIT_RESPONSE`: whether the output explicitly states whether the code is malicious.
- `LLM_MALICIOUS_RESULT`: whether the output classifies the code as malicious.
- `LLM_REASON_CONSISTENT`: whether the explanation is consistent with the model's binary decision.
- `LLM_REASON_CONSISTENT_WITH_CODE`: whether the explanation is grounded in the actual code rather than fabricated details.

### How annotate_sample.py Works

The module contains three main workflows:

- `gpt_auto_label(...)`: loads raw LLM outputs from `results/`, loads sample metadata from `malicious_packages_benchmark/`, sends the code plus the LLM output to GPT-5.2, and saves a JSON label file under `results_annotation/`.
- `mannual_verify_gpt_label(...)`: randomly samples 20 GPT-labeled results, displays the code and model output, and asks a human reviewer whether the GPT label is correct.
- `iter_samples_result(...)`: supports fully manual labeling without GPT by showing each sample and asking the reviewer to enter `y` or `n` for each metric.

For each sample, `gpt_auto_label(...)`:

1. Loads the target code from the metadata field such as `code_without_comment` or `filepath`.
2. Truncates the code to the first 10,000 characters.
3. Reads the corresponding raw LLM output from the model result JSON.
4. Marks `[ERROR]` outputs as invalid responses.
5. Sends the code and LLM output to GPT-5.2 with a fixed evaluation prompt.
6. Parses the GPT response as a JSON-like dictionary.
7. Saves progress after processing, so interrupted runs can resume.

Before running automatic verification, configure the OpenAI-compatible client in `annotate_sample.py`:

```python
client = OpenAI(
    api_key="sk-xxx",
    base_url="http://xxx"
)
```

Then update the `if __name__ == '__main__':` block for the experiment you want to annotate. The key variables are:

```python
experiment = 'code_mixture'
result_json_dir = f'results/{experiment}/zero-shot'
annotate_save_dir = f'results_annotation/{experiment}/zero_shot'
samples_info_file = 'malicious_packages_benchmark/inject_samples_info.json'
code_key = 'filepath'
```

For real-world malicious samples, use:

```python
experiment = 'real_world/malicious_samples'
result_json_dir = 'results/real_world/malicious_samples/zero-shot'
annotate_save_dir = 'results_annotation/real_world/malicious_samples/zero_shot'
samples_info_file = 'malicious_packages_benchmark/malicious_code_info_merge_with_typelabel.json'
code_key = 'code_without_comment'
```

For real-world benign samples, use:

```python
experiment = 'real_world/benign_samples'
result_json_dir = 'results/real_world/benign_samples/zero-shot'
annotate_save_dir = 'results_annotation/real_world/benign_samples/zero_shot'
samples_info_file = 'malicious_packages_benchmark/benign_code_info.json'
code_key = 'filepath'
```

For obfuscation samples, use:

```python
experiment = 'obfuscation'
result_json_dir = 'results/obfuscation/zero-shot'
annotate_save_dir = 'results_annotation/obfuscation/zero_shot'
samples_info_file = 'malicious_packages_benchmark/obfuscated_samples_info.json'
code_key = 'filepath'
```

For code-mixture samples, use:

```python
experiment = 'code_mixture'
result_json_dir = 'results/code_mixture/zero-shot'
annotate_save_dir = 'results_annotation/code_mixture/zero_shot'
samples_info_file = 'malicious_packages_benchmark/inject_samples_info.json'
code_key = 'filepath'
```

After configuring the block, run:

```bash
python annotate_sample.py
```

The generated annotation files are saved under `results_annotation/<scenario>/<prompt_mode>/` with names such as:

```text
gpt_label_code_mixture_qwen3-8b.json
```

## Computing Metrics

After raw outputs are annotated, run the corresponding statistics scripts:

```bash
python statistics_llm_results_realworld_malicious.py
python statistics_llm_results_realworld_benign.py
python statistics_llm_results_obfuscation.py
python statistics_llm_results_codemixture.py
python statistics_mannual_verify.py
```

These scripts read from `results_annotation/` and write CSV summaries to `results_evaluate_metric/`.

## Notes

- Run commands from the repository root unless a subdirectory README states otherwise.
- Some scripts currently use fixed paths and experiment settings in their `if __name__ == '__main__'` blocks. Adjust those variables before running a different scenario.
- Raw LLM outputs, GPT annotations, and final metrics are stored separately so each stage can be rerun independently.
